## Summary
Implement backup creation logic that copies all config files to a timestamped backup directory before any destructive update operations. This provides the safety net for rollback if updates fail.

## Background
Config updates can fail mid-process due to Docker not running, disk full, permission errors, or other issues. We must backup existing configs before any destructive operations so users can recover their working configuration. This is critical for user trust and safety - a failed update should never leave the system in a broken state.

This ticket implements the first critical step of the safe update process: creating a complete backup of the current configuration state.

Reference: `architecture.md` lines 114-169 for the complete update process with backup as step 1.

## Acceptance Criteria

- [ ] User .env file is backed up if it exists (optional file)
- [ ] Backup directory has permissions set to 0o700
- [ ] All backed-up files have permissions set to 0o600
- [ ] Function returns backup directory path on success

## Technical Requirements
- Use ISO 8601 timestamp for backup directory name, sanitized for filesystem (replace colons with dashes)
- Check file type with `fs.lstatSync()` to prevent symlink attacks (must be regular file)
- Use `fs.promises.copyFile()` for async file copying
- Handle missing optional files gracefully (.env may not exist)
- Create backup directory with `fs.promises.mkdir(dir, { recursive: true, mode: 0o700 })`
- Files to backup:
  * docker-compose.yml (required)
  * init.sql (required)
  * Dockerfile.mcp-server (required)
  * .maproom-version (required)
  * .env (optional - preserve user environment variables)

## Implementation Notes
**Module Location:**
- Modify: `packages/maproom-mcp/src/config-manager.ts`
- Add function: `backupConfigs(): Promise<string>`

**Backup Flow:**
1. Generate timestamp: `new Date().toISOString().replace(/:/g, '-')`
2. Create backup directory: `~/.maproom-mcp/backups/YYYY-MM-DDTHH-mm-ss.sssZ/`
3. For each config file:
   - Check if file exists
   - Verify it's a regular file (not symlink) with `lstatSync()`
   - Copy to backup directory
   - Set permissions to 0o600
4. Return backup directory path

**Error Handling:**
- If required file missing → throw error with file path
- If symlink detected → throw security error
- If disk full → throw error with disk space message
- If permission denied → throw error mentioning permissions

**Security Considerations (from `security-review.md`):**
- Symlink attack prevention: Use `fs.lstatSync()` to verify regular files before copying
- Path traversal: Sanitize timestamp before using in path (remove path separators)
- Permissions: Set 0o700 for directory, 0o600 for files immediately after creation
- Reference: `security-review.md` lines 188-236 for backup security requirements

## Risk Assessment
- **Risk**: Symlink attacks - attacker replaces config file with symlink to sensitive system file
  - **Mitigation**: Use `lstatSync()` to verify file is regular file, not symlink
  - **Severity**: High (could expose sensitive files)

- **Risk**: Disk space full - backup fails during copy
  - **Mitigation**: Check disk space before starting, provide clear error message
  - **Severity**: Medium (update fails but no data loss)

- **Risk**: Path traversal - timestamp contains path separators
  - **Mitigation**: Sanitize timestamp by replacing colons, validate final path
  - **Severity**: Medium (could write outside backup directory)

**Important Note About .env File**:
The .env file is **not** generated by the package. It may be created manually by users to override environment variables. The backup logic must:
- Check if .env exists before attempting to back it up
- Continue successfully even if .env is missing (it's optional)
- Only backup .env if it exists in the cache directory
